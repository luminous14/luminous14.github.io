Comparing Modules and Classes

Before researching ruby modules for this blog post, I had a vague notion that they can add functionality to my code. Reading through chapter 4 of "The Well-Grounded Rubyist" today has helped shed some light on what they do in Ruby and how they compare to classes.

The definition of module from dictionary.com reads:

module
a separable component, frequently one that is interchangeable with others, for assembly into units of differing size, complexity, or function.
The term module is used aptly in Ruby, because they are components that can be inserted into classes interchangeably to create new designs.

Here's a run down of some of the things I've learned:

  1. A class is a type of module.


  2. Modules are structured the same as classes, except when defining a module, we use the keyword "module" instead of "class".


  3. We can create an instance of a class, but not an instance of a module. In my post last week I used an example of a BoxOfChocolates class where I can create as many kinds of chocolate boxes as my heart desires. These chocolate boxes are each an instance of the BoxOfChocolates class. Each of my boxes can execute the instance methods defined within the BoxOfChocolates class. While you can't create an instance of a module, you can mix a module in with a class. This means I define a set of methods within a module, and then source it into my class later. The methods from the module are used within my class along with the instance methods defined by my class. To source a module into a class we use the keywords "include" or "prepend". "include" is used when we want class methods to have priority and "prepend" is used when we want module methods to have priority. This is important when a module and a class contain methods that have the same names.


  4. A class inheriting methods from another class is similar to a class having methods mixed in from a module. The big difference is that a class can only inherit methods from one other class, but it may have any number of methods from any number of modules mixed into it.


  5. Conventionally, classes are named with a noun and modules are named with an adjective, although this is not a rule. An example might be the following classes: BoxOfChcolates, BagOfSkittles, BucketOfGumballs. These classes might need to make use of the same methods, so I create a module called Sugary containing those methods to be mixed in with my classes. A module is concerned with what behaviors we add to a class. A class is about what kind of object we are creating, which will include the behaviors of the module. Like an adjective describing a noun, a module is used to describe a class.


  6. Modules can contain classes. To access a class from within a module you would use this syntax:

    instance = Module::Class.new

    or

    box = Sugary::ChocolateBox.new
I've heard the term module numerous times and am happy to have a better understanding of what they are and how they can be used. Now I can make start making other candies besides chocolates all the more easily.
